# SPEC.md

# Project: Lattice
## Purpose
Lattice is a graph-based work tracking system designed to replace traditional tools like Jira. It uses a fixed data model and a strict three-state lifecycle to keep the user-facing interface simple while modeling complex development workflows through relationships and views.

---

# Core Principles

1. **Simplicity at the Interface**
   - Users interact with six concepts: WorkItems, States, Tags, Relationships, Hierarchy, and Views
   - No configurable workflows
   - The data model is fixed and has no user-extensible schema

2. **Graph-Based Model**
   - Work is represented as a graph, not a rigid hierarchy
   - Relationships are queryable, have globally unique IDs (UUID v4), and are dependent on WorkItems for referential integrity

3. **Strict Lifecycle**
   - Only three states exist:
     - NotDone
     - InProgress
     - Completed

4. **Separation of Concerns**
   - State = lifecycle
   - Tags = metadata (no behavioral effect)
   - Relationships = structure

| Term         | Definition                          |
|-------------|-------------------------------------|
| WorkItem    | Fundamental unit of work            |
| State       | Lifecycle stage                     |
| Tag         | Metadata label                      |
| Relationship| Directed link between WorkItems     |

# Data Model

## WorkItem

```json
{
  "id": "string (UUID v4)",
  "title": "string (max 500 characters)",
  "description": "string (max 10000 characters)",

  "state": "NotDone | InProgress | Completed",

  "tags": ["string (max 100 characters each, no commas, max 50 tags)"],

  "type": "string (max 100 characters, optional)",

  "parent_id": "string (UUID v4, optional, must reference a valid WorkItem ID other than its own ID, or be null)",

  "relationships": [
    {
      "id": "string (UUID v4)",
      "type": "blocks | depends_on | relates_to | duplicate_of",
      "target_id": "string (UUID v4)"
    }
  ],

  "created_at": "string (ISO 8601 with timezone, e.g. 2026-04-07T12:00:00Z)",
  "updated_at": "string (ISO 8601 with timezone, reflects last modification, always >= created_at)"
}
```

Invariants:
- `created_at` is immutable after WorkItem creation.
- `updated_at` is automatically updated on any modification and must always be >= `created_at`.

---

## Constraints

- `id` must be a UUID v4, generated by the system (applies to both WorkItem IDs and relationship IDs)
- `state` must be one of the three allowed values
- `relationships` must reference valid WorkItem IDs
- `parent_id` must reference a valid WorkItem ID (not its own ID) or be null
- Circular dependencies are allowed, including self-referential relationships (where `target_id` is the owning WorkItem's ID); the system must provide a query endpoint that detects and returns cycles in the dependency graph on demand
- `type` is NOT enforced by system logic — any string value is accepted
- `tags` are free-form strings with no system-defined semantics; commas are not allowed
- Relationships are unique per combination of (owning WorkItem ID, target_id, type)

---

## State Model

### Allowed Transitions

NotDone -> InProgress -> Completed

### Rules

- No backward transitions by default
- Backward transitions are allowed when the request includes an `"override": true` field and the caller has admin role (determined by an `X-Role: admin` request header). The `X-Role` header is set by a trusted upstream component (API gateway or reverse proxy) that strips any client-supplied `X-Role` header before injecting its own. The system trusts this header unconditionally; it does not perform its own authentication or role verification. If the upstream component fails and allows a client-supplied `X-Role` header through, the system will honor it — securing the header pipeline is the responsibility of the deployment infrastructure, not this system. Any `X-Role` value other than `admin` (including an empty string value), or absence of the header, is treated as non-admin. If multiple `X-Role` headers are present, only the first occurrence is considered. Non-admin callers submitting `override` receive a `403 Forbidden` error. The `override` field is only meaningful for backward transitions; for forward transitions it is ignored regardless of role. If `override` is `false` or omitted, default transition rules apply.
- State transitions must be explicit (no automatic transitions)

## Relationships

### Supported Types

| Type         | Description                      |
|-------------|----------------------------------|
| blocks      | This item blocks another         |
| depends_on  | This item depends on another     |
| relates_to  | Informational link               |
| duplicate_of| Marks duplicate work             |

Hierarchy is handled exclusively by `parent_id`, not by relationship types.

### Rules

- Relationships are directional
- System must support reverse lookup (e.g., given a target, find all sources)
- Referential integrity is enforced: the owning WorkItem (the WorkItem containing the relationship) and the `target_id` must both reference existing WorkItems. If the owning WorkItem (path resource) does not exist, `404 NOT_FOUND` is returned. If the `target_id` references a non-existent WorkItem, `422 VALIDATION_ERROR` is returned. Reverse lookup is supported via the `relationship_target_id` query parameter on the `List WorkItems` API.

---

## Tags

### Characteristics

- Free-form strings (max 100 characters each, commas are not allowed)
- No schema enforcement
- Maximum 50 tags per WorkItem

### Examples

- blocked
- delayed
- needs-review
- high-priority

### Rules

- Tags must NOT affect state transitions
- Tags must NOT be treated as workflow (the system must not use tags to enforce sequences of operations, conditional logic, or to derive system behavior)
- Tags have no system-defined behavioral semantics; they are purely informational metadata

---

## Hierarchy

Hierarchy is implemented exclusively via `parent_id`.

### Rules

- Only one parent allowed per WorkItem
- Arbitrary depth allowed, up to a maximum of 100 levels. Attempts to create or update a WorkItem that would exceed this limit must return a `422 VALIDATION_ERROR`.
- Circular parent-child relationships are forbidden (e.g., A parent of B, B parent of A). The system must detect and reject any `parent_id` update that would create a cycle, returning `422 VALIDATION_ERROR`.
- No enforced hierarchy types (e.g., Epic/Task)
- Deleting a parent WorkItem does not cascade delete children; the system automatically sets child WorkItems' `parent_id` to null.

---

## Views

### Board View

- Columns based on state:
  - NotDone
  - InProgress
  - Completed

### List View

- Filterable by:
  - tags
  - type
  - state
  - parent_id

### Graph View

- Displays relationships between WorkItems
- Shows dependencies and blockers
- Data for this view is provided by `List WorkItems` (which returns each WorkItem's relationships array) and `Detect Cycles` APIs

---

## Query Capabilities

System must support filtering by:

- state
- tags
- type
- relationships (via `relationship_type` and `relationship_target_id` query parameters; when both are provided, they must match the same relationship instance)
- parent_id

Example:

```
state = InProgress AND tag = blocked
```

## API Specification

All request and response bodies use JSON. Timestamps are ISO 8601 with timezone (UTC).

### Error Response Format

All error responses use the following structure:

```json
{
  "error": {
    "code": "string (machine-readable error code)",
    "message": "string (human-readable description)"
  }
}
```

Common error codes:

| HTTP Status | Code                  | Description                           |
|-------------|-----------------------|---------------------------------------|
| 400         | INVALID_INPUT         | Malformed or invalid request body     |
| 404         | NOT_FOUND             | WorkItem or relationship not found    |
| 409         | INVALID_TRANSITION    | State transition not allowed          |
| 403         | FORBIDDEN             | Override requested without admin role |
| 422         | VALIDATION_ERROR      | Referential integrity violation       |

### Create WorkItem

`POST /workitems`

Request:

```json
{
  "title": "string (required, max 500 characters)",
  "description": "string (optional, defaults to empty, max 10000 characters)",
  "type": "string (optional, max 100 characters)",
  "tags": ["string (max 100 characters each, no commas, max 50 tags)"],
  "parent_id": "string (optional, must be a valid WorkItem ID other than its own ID)"
}
```

State defaults to `NotDone` and cannot be set on creation. Newly created WorkItems always have an empty `relationships` array. The `id`, `created_at`, and `updated_at` fields are system-generated; if provided in the request, a `400 INVALID_INPUT` error is returned.

Errors:
- `400 INVALID_INPUT` if `title` is missing, empty, or exceeds 500 characters; if `description` exceeds 10000 characters; if `type` exceeds 100 characters; if any tag exceeds 100 characters, contains a comma, or more than 50 tags are provided; if `id`, `created_at`, or `updated_at` are included
- `422 VALIDATION_ERROR` if `parent_id` references a non-existent WorkItem or the WorkItem's own ID

Response (`201 Created`):

```json
{
  "id": "string (UUID v4)",
  "title": "string",
  "description": "string",
  "state": "NotDone",
  "type": "string | null",
  "tags": ["string"],
  "parent_id": "string | null",
  "relationships": [],
  "created_at": "string (ISO 8601 with timezone)",
  "updated_at": "string (ISO 8601 with timezone)"
}
```

---

### Update WorkItem

`PATCH /workitems/{id}`

Request (all fields optional; only provided fields are updated; providing an empty `tags` array clears all tags):

```json
{
  "title": "string (max 500 characters)",
  "description": "string (max 10000 characters)",
  "state": "NotDone | InProgress | Completed",
  "type": "string | null (max 100 characters)",
  "tags": ["string (max 100 characters each, no commas, max 50 tags)"],
  "parent_id": "string | null",
  "override": "boolean (optional, requires admin role for backward state transitions)"
}
```

The `id`, `created_at`, and `updated_at` fields are system-generated and must not be included in the request.

Response (`200 OK`): Full WorkItem object, including current relationships.

Errors:
- `400 INVALID_INPUT` if any field exceeds its maximum length, tag constraints are violated (including commas), or system-generated fields (`id`, `created_at`, `updated_at`) are included
- `404 NOT_FOUND` if WorkItem does not exist
- `409 INVALID_TRANSITION` if state transition is not allowed
- `403 FORBIDDEN` if `override` is true but caller is not admin
- `422 VALIDATION_ERROR` if `parent_id` references a non-existent WorkItem or the WorkItem's own ID

---

### Get WorkItem

`GET /workitems/{id}`

Response (`200 OK`): Full WorkItem object, including current relationships.

Errors:
- `404 NOT_FOUND` if WorkItem does not exist

---

### List WorkItems

`GET /workitems`

Query Params (all filters are combined with AND logic):

- `state` — filter by state value (allowed values: `NotDone`, `InProgress`, `Completed`)
- `tags` — filter by tag (comma-separated values in a single parameter, e.g., `tags=tag1,tag2`; items must have all specified tags)
- `type` — filter by type value
- `parent_id` — filter by parent WorkItem ID (must be a valid UUID v4)
- `relationship_type` — filter to WorkItems that have a relationship of this type (allowed values: `blocks`, `depends_on`, `relates_to`, `duplicate_of`)
- `relationship_target_id` — filter to WorkItems that have a relationship targeting this ID (must be a valid UUID v4)
  - When both `relationship_type` and `relationship_target_id` are provided, they must match the same relationship instance (i.e., the WorkItem must have at least one relationship of the specified type targeting the specified ID)
- `is_blocked` — filter to WorkItems that are blocked (have at least one `depends_on` target not in `Completed` state). Accepts `true` or `false`.
- `is_ready` — filter to WorkItems that are ready (all `depends_on` targets are in `Completed` state). Accepts `true` or `false`.
- `page` — page number (default: 1, must be a positive integer; `400 INVALID_INPUT` if invalid)
- `page_size` — items per page (default: 50, must be a positive integer between 1 and 200 inclusive; `400 INVALID_INPUT` if invalid)

Response (`200 OK`):

```json
{
  "items": [WorkItem],
  "total": "integer",
  "page": "integer",
  "page_size": "integer"
}
```

---

### Add Relationship

`POST /workitems/{id}/relationships`

Request:

```json
{
  "type": "blocks | depends_on | relates_to | duplicate_of",
  "target_id": "string (UUID v4, must be a valid WorkItem ID)"
}
```

Response (`201 Created`):

```json
{
  "id": "string (UUID v4)",
  "type": "string",
  "target_id": "string"
}
```

Errors:
- `404 NOT_FOUND` if the source WorkItem (path `{id}`) does not exist
- `422 VALIDATION_ERROR` if the `target_id` references a non-existent WorkItem
- `400 INVALID_INPUT` if type is not a supported relationship type, or if a system-generated field like `id` is included in the request
- `422 VALIDATION_ERROR` if a duplicate relationship already exists (uniqueness is determined by the combination of source WorkItem ID, target_id, and type)

---

### Remove Relationship

`DELETE /workitems/{id}/relationships/{rel_id}`

Response: `204 No Content`

Errors:
- `404 NOT_FOUND` if the WorkItem specified by `{id}` does not exist, or if no relationship with the given `{rel_id}` exists on that WorkItem (relationship IDs are globally unique, but the relationship must belong to the specified WorkItem)

---

### Delete WorkItem

`DELETE /workitems/{id}`

Deleting a WorkItem is an atomic operation. It deletes the WorkItem, all relationships where this WorkItem is the owning WorkItem (source), and all relationships where this WorkItem is the `target_id` of another WorkItem's relationship. Child WorkItems that reference this WorkItem via `parent_id` have their `parent_id` automatically set to null. If any part of this operation fails, the entire transaction must be rolled back, leaving the system in its prior consistent state.

Response: `204 No Content`

Errors:
- `404 NOT_FOUND` if WorkItem does not exist

---

### Detect Cycles

`GET /workitems/{id}/cycles`

Returns all unique dependency cycles that contain the specified WorkItem. Cycle detection treats both `depends_on` and `blocks` as directed edges in the same dependency graph (A `depends_on` B and A `blocks` B are both traversed as A -> B).

Response (`200 OK`):

```json
{
  "cycles": [
    ["WorkItem ID", "WorkItem ID", "..."]
  ]
}
```

Returns an empty `cycles` array if no cycles are found.

Errors:
- `404 NOT_FOUND` if WorkItem does not exist

## Derived Behavior

### Block Detection

A WorkItem is considered "blocked" if:

- It has at least one `depends_on` relationship
- AND that relationship's target is NOT in `Completed` state

---

### Completion Readiness

A WorkItem can be flagged as "ready" if:

- All of its `depends_on` targets are in `Completed` state (a WorkItem with no `depends_on` relationships is considered ready)

## Non-Goals

- No custom workflows
- No role-based complexity (initially, beyond admin override for state transitions)
- No required field explosion
- No rigid type enforcement

---

## Acceptance Criteria

1. User can create, update, and view WorkItems
2. User can assign tags freely
3. User can define relationships between items
4. System enforces only 3 states with forward-only transitions (admin override for backward)
5. System supports hierarchical parent-child relationships via `parent_id`
6. System supports querying/filtering by state, tags, type, relationships, and parent_id
7. System provides `List WorkItems`, `Get WorkItem`, and `Detect Cycles` APIs to enable board, list, and graph views
8. No configurable workflows exist
9. All API endpoints return structured error responses for invalid operations

---

## Risks

- Users may attempt to recreate Jira via tags/types
- Graph complexity may become difficult to visualize
- Lack of strict structure may confuse new users

---

## Guiding Constraint

The system must never add features that increase user-facing concepts (beyond WorkItems, States, Tags, Relationships, Hierarchy, and Views) to solve internal modeling problems.

All complexity must be absorbed by:

- data model
- system logic
- views

NOT the user interface.
